Triage tool: Settings → Advanced → "Inspect asset by filename".

End-user case after build 118 — 19 items still in quarantine that don't fit any of the divergence axes we've now covered (115/116/117/118). Rather than keep iterating on speculative axes, this build adds a debug action so any future case can be triaged from the actual data instead of from guesses.

Settings → Advanced → "Inspect asset by filename" opens an alert with a text field. Enter a filename (e.g. `IMG_1234.MOV`) → tap Inspect → cairn dumps a side-by-side view to the persistent log:

- Every phone-side PHAsset matching the filename case-insensitively on either `PHAsset.value(forKey: "filename")` OR any of its PHAssetResource `originalFilename` values. Per-match row: `localId`, KVC filename, per-resource type+filename list, `creationDate` (ISO8601 with fractional seconds + integer seconds), modification seconds, `isHidden`, source type, `mediaSubtypes.photoLive` flag.
- Every Immich server asset matching the filename. Per-match row: id, `originalFileName`, `fileCreatedAt` (ISO8601 + integer seconds), checksum, `livePhotoVideoId`.
- Verdict per server match: whether the engine's *current* alive-on-phone filter (with ±1s tolerance) would suppress it.

The fetch enumerates all source types and hidden assets, so a Hidden/CloudShared/iTunesSynced match isn't accidentally excluded. The server query goes through `search/metadata` with `originalFileName` filter; bounded to 1000 results (per-asset triage, not full enumeration).

Output lands in the persistent log (build 113), which survives process restarts, so the next diagnostic export shows the side-by-side. To use: install 119 → Settings → Advanced → Inspect asset by filename → enter the filename of a stuck quarantine entry → Settings → Advanced → Export diagnostic logs → share.

No behavior change to sync, indexing, or reconciliation.
