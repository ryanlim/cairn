Cross-launch diagnostic-log capture so we stop losing failure context to the per-process OSLog scope.

End-user diagnostics were repeatedly turning up empty during the most important moment to triage — after the very failure we needed to see. iOS sandboxes third-party apps to `OSLogStore.scope(.currentProcessIdentifier)`, which means every relaunch wipes the prior process's log entries from anything the export can read. The "sync starts, gets killed by background suspension or memory pressure, the next launch's export shows nothing of value" pattern was eating our ability to diagnose the multi-process failure modes that 100k+ libraries seem to land in.

This build adds a persistent on-disk log capture:

- `PersistentLogStore` — actor managing a rolling 5 MB primary file + 1 rolled file in the app's `Documents/Diagnostics/` directory. Two-file max → ~10 MB worst-case disk usage. Excluded from iCloud backup (it's diagnostic noise, not user data).
- `DiagnosticLogFlusher` — main-actor coordinator started in `CairnApp.init`. Polls `OSLogStore` every 20 seconds for entries on cairn's subsystems (`app.cairn` and `app.cairn.ios`), formats them, and appends to the persistent store. Also fires explicitly on scene-phase → background so the tail of foreground activity is captured before iOS suspends.
- `LogExporter.export()` — now reads the persistent file first (rolled then live, chronological) and prepends it to the current-process buffer block. The export file gains two delimited sections: "persistent capture (prior + current launches)" and "current-process buffer (newest, may overlap persistent)".

Result: a tester whose initial-sync attempts get killed across multiple launches can still send a single export file that captures activity from every one of those launches, not just the most recent. Worst-case data loss on a sudden iOS kill is ~20 seconds (the periodic poll interval); a normal background→terminate flow captures everything because the on-background flush runs first.

No changes to sync, indexing, or reconciliation behavior. The persistent store is dormant on a fresh install until the first 20-second poll fires.
