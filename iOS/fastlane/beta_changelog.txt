Inspect asset by filename: two-pass strategy so it finishes in seconds instead of minutes.

Build 119's `inspect-asset-by-filename` was unusable in practice on real libraries: the action called `PHAssetResource.assetResources(for: asset)` for every PHAsset (~30 ms each), so a 7k library took ~3.5 minutes to complete and a 100k library took close to an hour. Long enough that any export taken before completion shows only the `inspect: starting` line, with no actual triage data captured.

Two-pass strategy now:

- **Pass 1 — KVC filename only.** `PHAsset.value(forKey: "filename")` is a pure property access (~0.1 ms per asset, no resource enumeration). A 7k library finishes the KVC sweep in under a second; a 100k library in ~10 seconds. Catches the common case where the asset's name agrees with the server's `originalFileName`.
- **Pass 2 — resource-level scan.** Runs only when Pass 1 yields zero matches. This is the rare divergence case where the phone-side asset has a different KVC filename from the resource that was uploaded. Slow (the old behavior) but bounded by the rarity.

Per-pass progress logging lets the user see the action making progress: KVC pass logs every 20k assets, resource pass logs every 5k assets with a running match count. So an export taken mid-run shows "KVC pass at 80000/108000" instead of silence.

No change to the data captured per match — same side-by-side phone-vs-server dump, same verdict against the engine's current alive-on-phone filter (with ±1 s tolerance). Just dramatically less wall-clock waiting before the data lands in the log.
