Alive-on-phone safety: include hidden + iCloud-shared + iTunes-synced assets in the alive set.

Spot-check after build 115 still flagged a video that's confirmed alive in Photos.app — same filename, same timestamp on both phone and Immich. Diagnosis: the PHAsset fetch that builds the alive-key set was filtering to `.typeUserLibrary` only and excluding hidden assets. An older video that ends up in iCloud Shared Library or was synced via iTunes/Finder historically would slip past the filter, the alive-key set wouldn't see it, the engine's `wouldBeCandidates` filter would let it through, and quarantine.

Other cairn paths (notably orphan sweep at `PhotoKitPersistentChangeReconciler.swift:1190`) already include `.typeCloudShared` — the filter narrowness was specific to this fetch site. This build broadens both:

- `visibleFetch` in `performLiveReconciliation` — `includeHiddenAssets = true`, `includeAssetSourceTypes = [.typeUserLibrary, .typeCloudShared, .typeiTunesSynced]`. Same downstream uses (liveLocalIds, editRetirementHeld, alive-key set) — all benefit from the broader view.
- Imputation's PHAsset enumeration in `runImputationPass` — same broadening. Keeps the piggyback metadata write current for hidden/shared/synced assets too, so the alive-key set's metadata-store source (build 115) catches the filename-source-divergence cases for these asset classes as well.

No behavior change for deletion proposals already in progress — these source-types still can't be deleted from Photos.app, so cairn doesn't propose to trash their server counterparts either. The change is strictly in the "is this still on phone" safety direction.
