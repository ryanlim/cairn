Partial revert of build 125's enumeration unification.

End-user reported "sync hanging" on 125 with an existing-cache (no wipe) install. Log analysis showed the reconciler's untracked sweep firing on 524 previously-invisible assets:

```
[cairn.recon] untracked sweep: 524 PHAssets visible but neither indexed
              nor queued — adding to insert pipeline
```

Followed by a few hundred `[cairn.hash] slow-asset: id=... took=X — probably iCloud fetch` lines at 2-60 sec each. So the "hang" was the hashing pipeline trying to iCloud-download and hash 524 newly-visible Cloud-Shared + iTunes-synced + hidden assets. Working as designed under build 125's filters — but the design was wrong.

Build 125 conflated two enumeration purposes:

- **Safety/suppression filters** (alive-on-phone, orphan sweep): SHOULD be broad. Never falsely propose to trash a server entry whose phone-side asset is still alive in any form. Goes through every PhotoKit source type, includes hidden.
- **Active management** (hashing, untracked discovery): SHOULD be narrow. Only manage assets the user can actually delete via Photos.app. iTunes-synced and Cloud-Shared assets can't be deleted by the user, so cairn shouldn't be hashing them as deletion-tracking candidates. Hidden assets currently excluded (privacy default; revisit later).

This build narrows the two active-hashing sites back down:

- `hashAllCurrentAssets` full-enum fetch: `.typeUserLibrary` only, hidden=false
- `discoverUntrackedAssets` live-id sweep: `.typeUserLibrary` only, hidden=false

Keeps the two safety sites broadened from 125:

- `runIncremental` id-existence check: stays at `[.typeUserLibrary, .typeCloudShared, .typeiTunesSynced]` + hidden=true (right call — false-positive "deleted" reports for shared/synced/hidden are exactly what the broader filter prevents)
- `reconcileCacheAgainstLibrary` orphan sweep: stays at the same broader filter (right call — false-positive orphan detection is the harm being prevented)

The 7502 vs 6946 "discrepancy" your last log surfaced is actually correct — imputation enumerates the broader set (for alive-on-phone metadata), full-enum hashes the narrower set (only manageable assets). Those numbers diverging by source-type-excluded-asset count is the desired state.

For installs that hit 125 with the untracked-sweep fire (you), the in-flight hashing of those 524 assets should be interrupted on 126 install — `discoverUntrackedAssets` no longer returns them. The partial hashes that did complete stay in cache (no harm), the ones that didn't, won't be re-attempted.

Apologies for the cycle. 125 was the right structural insight (safety vs active mgmt should diverge) but the wrong direction (broadened the active-mgmt path instead of narrowing it).
