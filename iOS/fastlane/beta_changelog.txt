Fix: alive-on-phone safety check no longer misses assets when filename source diverges.

End-user report on build 113: a video confirmed still in their Photos library landed in quarantine, with matching filename and capture date — exactly the case the build-102 alive-on-phone filter was designed to suppress. Diagnosis exposed a brittleness: cairn was building the alive-key set from one filename source and matching against another.

- Imputation enumeration uses `PHAssetResource.selectPrimaryResource(from:)?.originalFilename` (resource payload — what Immich's mobile uploader sends to the server)
- Build-102's alive-key pass used `PHAsset.value(forKey: "filename") as? String` (PHAsset internal KVC)

These usually agree but can diverge on the same asset — most often a case shift (`IMG_1234.MOV` vs `img_1234.MOV`), occasionally content. When they diverge, the exact-match Set lookup in the engine misses, the asset is wrongly classified as a candidate, and lands in quarantine.

Two composing fixes:

1. **`AlivePhoneAssetKey` constructor normalizes filename to lowercase.** Case-only differences fall away regardless of which source produced the key. The struct's stored `filename` is now always lowercased. 2 new engine tests cover the normalization (case-insensitive equality + stored-form check).

2. **Alive-key build in `performLiveReconciliation` populates from BOTH sources.** PHAsset KVC pass (existing, fast) plus a `LocalAssetMetadataStore` pass (uses the same resource-level filename that imputation joins on). The `Set` de-dups. Any asset whose phone-side filename matches the server's stored `originalFileName` via either path is now recognized as alive.

Net effect: the engine's alive-on-phone filter is robust to the filename-source divergence we've observed. The 41 quarantine entries the tester reported will need to be reviewed once on the new sync (existing stamps in ConfirmedDeletedStore persist; the fix prevents new instances), then dismissed via Pending Review.
