Incremental server sync (opt-in): a new Settings → Advanced toggle, "Incremental server sync," switches the server-side discovery pass from paginated rescans to Immich's sync/stream change-data-capture endpoint. On a healthy 50k-asset library the steady-state sync drops from ~200 pagination round-trips to a single streaming call returning only the events since the last acknowledgment — typically zero to tens of events for a normal sync interval. Default off during initial roll-out; flip it on under Settings → Advanced once you've regenerated your API key with the new sync.* scopes (sync.stream, sync.checkpoint.read, sync.checkpoint.update, sync.checkpoint.delete). If the scopes are missing, cairn silently falls back to the paginated path — flipping the toggle on with an old key never blocks sync, it just keeps the old speed.

Under the hood: cairn caches every server asset in a per-(URL, userId) SwiftData store and advances both server-side and local cursors after each batch. The cache survives relaunches. Crash safety: events are written to the local cache before the server gets its ack, so a crash mid-sync replays at most one batch — the cache's apply step is idempotent by server-side asset id.

Setup screen now lists the four sync.* scopes alongside the existing seven so users regenerating their API key today won't need to do it again when the default flips on next release.
