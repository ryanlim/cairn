Build-121 migration: backfill `allResourceFilenames` for existing metadata rows so the alive-on-phone fix actually takes effect on prior installs.

End-user report after installing 121: zero stuck quarantine items resolved, multiple syncs run, engine output unchanged at `aliveOnPhone=6`. The build-121 fix was correct but only writes the new `allResourceFilenames` field via the imputation piggyback — which only runs on bootstrap syncs (token nil or imputation marker absent). For installs already past their initial imputation, the metadata rows stay with the empty default and the alive-key build can't see the non-primary resource filenames.

This build adds a one-shot per-partition migration invoked at bootstrap (`migrateResourceFilenamesIfNeeded`). The flow:

1. UserDefaults gate: `cairn.migration.resourceFilenames.<partition>`. Migration runs once per Immich account on this device.
2. Snapshot `LocalAssetMetadataStore`. Filter to rows where `allResourceFilenames` is empty.
3. `PHAsset.fetchAssets(withLocalIdentifiers:)` to batch-load every affected asset; PhotoKit returns them via one query.
4. Per-row: `PHAssetResource.assetResources(for: asset)` (~30 ms each) → collect every resource's `originalFilename` → rewrite the metadata row with the full list.
5. Log progress every 500 rows. Mark migrated on completion.

Runs as `Task.detached` so the bootstrap path doesn't block. Cost scales with row count needing backfill — a few minutes on a 10k library, ~half an hour on a 100k library. After completion, the next sync's alive-key build pulls from the now-populated metadata and the edited-asset class of quarantine entries silently suppresses.

If the migration is interrupted (device locked, app backgrounded, crash), the flag isn't set and it retries on the next bootstrap. Idempotent — re-runs only touch rows still missing the field.

Skips rows where the asset has no extra resources (just a primary) or is no longer fetchable from PhotoKit — no point rewriting for the no-op case.

End-user impact: install 122, leave cairn open briefly while the migration runs in the background (the progress logs show ~3-5 minutes on a 7k library), then trigger a sync. The 19 stuck quarantine items should drop on that next sync as the alive-key set picks up the now-populated resource filenames.
