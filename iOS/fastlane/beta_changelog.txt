Fix: the fast initial scan from build 85 was silently broken on current Immich. Immich removed the per-device asset identifier (deviceAssetId) from its asset schema in April 2026 as part of moving to checksum-based identity. With that field gone, the trust-seed join had nothing to match on — "Matching from server" would run and find zero hits, falling through to full local hashing.

This build pivots the join to a different stable pair: filename + capture date. For photos that Immich's mobile app uploaded from this iPhone, both values round-trip exactly through the upload → server → cairn path, so the match should land just as well as before. The join is heuristic where the previous one was rigorous, so the safety guards are tightened: any (filename, capture-date) pair that matches more than one non-trashed server row is treated as ambiguous and falls through to local hashing. The near-zero-hit fallback, the imputed flag, the verify-on-edit path, and the 30-day Immich Trash recovery window are all unchanged.

Worst case from a bad imputed match: cairn trashes the wrong server row when you've already asked for some delete. Recoverable from Immich Trash.

To test on an already-indexed install: Settings → Data & recovery → Export data (backup), then Library → Trust server checksums → ON, then Data & recovery → Clear hash cache → Clear, then sync. The "Matching from server" phase should now actually produce a non-zero trust-seeded count. Restore the backup via Data & recovery → Import data when done.
