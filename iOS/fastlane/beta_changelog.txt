More accurate ETA during initial sync — surfaces iCloud-throttling slowdowns instead of smoothing them away.

The prior ETA computed a single cumulative rate (`session elapsed / session work`) and used that to project remaining time. On real iPhone hashing passes the rate is bimodal:

- Locally-cached assets: ~10-30 ms each
- iCloud-pending assets needing download: 2-15 seconds each (100-1000× slower)

When a scan starts on cached assets and then hits iCloud throttling for the remainder, the cumulative rate stays anchored to the early fast samples and ETA grossly underestimates. From the most recent diagnostic export the slow-asset events show exactly this shape — many 5-15s `slow-asset` log lines mid-scan.

This build adds a recent-window rate (90 seconds of moving samples) computed from a rolling buffer of `(time, cumulative-hashed)` checkpoints. ETA is now `max(cumulativeETA, recentWindowETA)`:

- When conditions are stable, the two values are ~equal and behavior is unchanged.
- When recent rate is slower than cumulative (iCloud throttling engaging, asset composition skewing toward heavier files), the higher pessimistic value wins.
- When recent rate is faster (rare — the early phase is usually faster), cumulative wins, avoiding mid-scan optimism.

Sample buffer is dual-capped: time-filtered to 2× the window on read, sample-count-capped at 200 on write. Bounded memory regardless of emit rate.

The user-visible value still passes through the existing sample-and-hold + median-of-5 publishing layer, so it doesn't flicker on every progress tick — but the underlying rate it's smoothing now reflects current conditions instead of session-wide average.

No changes to sync, indexing, or reconciliation behavior.
