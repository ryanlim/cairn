Live progress for the server-fetch step of large-library syncs.

End-user report: a 100k+ Immich library sat on the "Preparing" stage for 16+ minutes with no visible activity. The `.preparing` phase silently runs the paginated `listAllAssets` call, which on a slow self-hosted Immich can take 100+ round trips (8-16 minutes is plausible for that library size).

This build:

- ImmichClient.listAllAssets gains an `onPageLoaded(Int)` callback (fired after each successful page) and an `expectedTotal: Int?` capacity hint to skip ~17 Array reallocations on a 100k fetch.
- A parallel `assetStatistics()` fetch populates the expected denominator before the first paginated page returns.
- InitialScanScreen now renders a "X / Y server assets fetched" line under the prep label whenever the count is non-zero. When the total is known, shows the proportional count; otherwise the open-ended counter alone — either way, motion replaces apparent freeze.

Diagnostic value: the line tells you exactly which subsystem is slow. If the count ticks at a steady rate but slowly, the bottleneck is Immich's pagination throughput. If the count is stuck at 0, the bottleneck is elsewhere (DNS, TLS, app launch, statistics endpoint, etc.) and we know to look there.

A follow-up build will add bounded-concurrency parallel page fetch (4-8x speedup potential when Immich's pagination is offset-stable) — wanted to ship the visibility fix first so we can verify the bottleneck is actually pagination throughput on real reports.
