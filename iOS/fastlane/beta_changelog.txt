Alive-on-phone safety: capture every PHAssetResource filename, not just the primary.

The remaining stuck-quarantine cases turned out to be edited assets. End-user inspect-tool run on build 120 produced unambiguous data for IMG_2999.MOV:

```
phone[0] kvc='80EE02BF-7741-41A8-9FDF-DE85F0784744.MOV'
         resources=[
           video='IMG_2999.MOV',
           adjustmentData='Adjustments.plist',
           fullSizePhoto='FullSizeRender.jpeg',
           fullSizeVideo='FullSizeRender.mov'
         ]
server[0] name='IMG_2999.MOV'
```

PhotoKit replaces the PHAsset's KVC `filename` with a UUID-style placeholder once an asset is edited, AND the primary resource picker (`PhotoKitPhotoEnumerator.selectPrimaryResource`) returns the rendered edit (`FullSizeRender.mov`). Immich's server entry retains the pre-edit upload filename (`IMG_2999.MOV`) because the asset was uploaded *before* the edit, and that filename now exists only on the `.video` resource ā€” which neither the KVC pass nor the metadata-store (primary-resource-sourced) pass was looking at.

This affects every edited video in the user's library, which is most of the remaining quarantine entries.

Fix:

- `StoredLocalAssetMetadata` gains `allResourceFilenamesCSV: String = ""` ā€” SwiftData lightweight migration (defaulted field, no manual step). Tab-separated to avoid collisions with system-rendered filenames that legitimately contain commas.
- `LocalAssetMetadata` value type gains `allResourceFilenames: [String]` with a custom Codable decoder that defaults to `[]` for older JSON payloads.
- Imputation enumeration in `runImputationPass` already calls `PHAssetResource.assetResources(for: asset)` to grab the primary's filename + size; now also collects every resource's `originalFilename` into a list and passes it through the `phoneEntries` tuple. Piggyback metadata write stores the full list. Zero additional PHAssetResource calls ā€” free relative to the existing per-asset enumeration cost.
- `backfillMetadataIfNeeded` (the fallback path for cache-resident-but-metadata-missing ids) does the same.
- Alive-key build in `performLiveReconciliation` now emits an `AlivePhoneAssetKey` for every entry in `allResourceFilenames` in addition to the primary. The engine's existing case-insensitive + Ā±1s tolerance lookup catches the match.

No behavior change for unedited assets ā€” their primary resource filename already covers the alive-on-phone case. Edited assets that previously couldn't be suppressed now suppress correctly on the next imputation run (the metadata store gets populated with the resource list during the imputation piggyback; existing rows still default to empty until they're re-observed during a future imputation).

Existing quarantine stamps in `ConfirmedDeletedStore` from prior runs persist until manually dismissed (they pre-date this fix). New runs after a sync that re-populates metadata won't re-create those stamps.
