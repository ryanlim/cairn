Big win for 100k+ libraries: eliminate the silent hour between Matching from server and Hashing.

End-user diagnostic from a 108,936-asset library showed Matching from server finishing in 8 minutes (good), then ONE FULL HOUR of zero log activity before the user backgrounded the app in frustration. Hashing never started.

Root cause: `recordFullEnumerationMetadata` was running 108,936 sequential `PHAssetResource.assetResources(for:)` calls — ~30ms each = 50+ minutes silent — to record the same filename/creationDate/size metadata that the imputation pass *already enumerated 8 minutes earlier and threw away*. Two complete passes over the entire photo library back-to-back.

This build piggybacks metadata recording on the imputation enumeration. Imputation already calls `PHAssetResource.assetResources` per asset to extract the filename for its (filename, capture-second) join key; capturing the file-size in the same loop and writing all five fields (localId, filename, creationDate, modificationDate, fileSize) to `LocalAssetMetadataStore` as a side-effect makes the subsequent full-enum metadata pass a near-no-op (the existing "skip ids already recorded" guard handles it). Net effect on the user's 108k library: the silent hour collapses to ~50 milliseconds.

Also adds periodic progress + heartbeat logging to `recordFullEnumerationMetadata` so this kind of multi-thousand-asset silent loop can never recur unnoticed:
- Start: "starting N PHAssetResource calls — full-enum hashing waits for this to complete"
- Every 5,000 assets: "progress N/M, rate X/s, eta Ys"
- Completion: "complete: N entries recorded in Y ms"

Plus a short-circuit log when the entire pass is skipped because imputation already covered everything.

For users with fast-initial-scan disabled (no imputation pass): the metadata loop still runs, but now logs progress so they can see it's working and roughly when it'll finish.
